# AI 学习项目工程化改进建议

## 当前状态

项目已跑通一个完整的 AI 对话 MVP：用户认证、会话管理、MySQL 持久化、SSE 流式输出、经营事件 Function Calling。但工程化程度仍处于 Demo 阶段，需要在结构、安全、可维护性上继续完善。

## 问题与改进方向

### 1. 后端架构脚本化

**现状：** 所有路由、业务逻辑、数据库操作集中在 `src/server.ts`，文件超过 350 行。

**问题：**
- 职责混杂，阅读和修改成本高。
- 新增接口容易破坏现有代码。
- 没有统一的错误处理和日志。

**改进：** 按职责拆分为三层：
- **Routes**：只负责路径映射和参数解析。
- **Controllers**：处理请求响应、调用 Service。
- **Services**：承载业务逻辑和数据操作。

### 2. 输入校验缺失

**现状：** 各路由中手工判断字段类型和长度。

**问题：** 校验逻辑分散、重复，容易遗漏。

**改进：** 引入轻量校验库（如 zod）统一校验请求体。

### 3. 安全加固

**现状：** JWT secret 简单、无速率限制、无密码强度校验。

**改进：**
- 使用强随机 JWT secret，生产环境从环境变量读取。
- 登录接口增加速率限制。
- 密码增加强度校验。
- CORS `origin: "*"` 仅适合开发，生产需限定域名。

### 4. 前端工程化

**现状：** 无 ESLint/Prettier、无测试、状态管理用全局 ref。

**改进：**
- 配置 ESLint + Prettier。
- 引入 Vitest 做单元测试。
- 统一错误处理和加载/空状态组件。

### 5. AI 工程化

**现状：** prompt 硬编码、history 限制简单、无 retry/timeout。

**改进：**
- prompt 外置化。
- 增加模型调用超时和重试。
- Function Calling 结果增加 schema 校验。

## 短期可落地的行动

1. **拆分 `src/server.ts`**（当前即可做）
2. **增加统一的错误处理和日志**
3. **配置 ESLint + Prettier**
4. **补充登录速率限制和密码强度校验**
5. **增加前后端基础测试**

## 下一步建议

优先完成 `src/server.ts` 的拆分，这是后续所有改进的基础。
